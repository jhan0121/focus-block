# Thinking 봉쇄 규칙 (T1~T7)

이 규칙은 모든 세션, 모든 작업에서 자동 적용된다.

## 위반 감지 시 즉시 수행할 행동

훅(`validate-thinking.sh`)이 PreToolUse 시점에 아래 패턴을 감지하면 도구 호출을 차단한다.
차단 후에는 아래 "올바른 대체 행동" 열을 보고 즉시 그 행동을 수행한다.

| 규칙 | 즉시 중단해야 하는 표현 | 올바른 대체 행동 |
|------|------------------------|----------------|
| T1 | thinking에서 영어 단어 사용 | 해당 thinking 문장 삭제 후 한국어로 재시작 |
| T2 | "사용자의 의도는~", "다시 생각하면~", "더 효율적인~", "의도를 재해석" | 사용자 지시 원문을 그대로 다시 읽고 그대로 이행한다 |
| T3 | "우선 ~만", "일단 ~부터", "간단하게", "샘플로", "큰 파일이라서" | 지시된 전체 범위를 처음부터 구현한다. 분리하거나 나누지 않는다 |
| T4 | "아마~", "보통~", "~겠지", "~이어 보인다", "~인 것 같다", "이론적으로" | Read/Grep 도구로 실제 파일을 확인한 후에만 판단한다 |
| T5 | 실제 확인 없이 "완료" 선언, "보고하고 수정할까요" | 변경 파일과 호출 파일의 앞뒤 흐름을 Read로 직접 확인한다 |
| T6 | 지시하지 않은 방법·대안을 thinking에서 고려, "이렇게 하면 어떨까" | 사용자가 명시한 방법만 사용한다. 대안 고려 자체를 중단한다 |
| T7 | 토큰 절약, 효율 대안, 빠른 방법 추론, "응답이 길어질 수 있으니" | 지시된 전체 내용을 그대로 출력한다. 길이 기준으로 줄이지 않는다 |

## 적용 순서 (매 작업 시작 전)

1. 사용자 지시 원문을 정확히 읽는다 (재해석 금지)
2. 지시된 작업 범위 전체를 파악한다 (축소 금지)
3. 필요한 파일 목록을 특정한다 (추측 금지, Read/Grep으로 실제 확인)
4. 왜/어떻게/무엇을/어떤 식으로 4가지 질문에 실제 파일 근거로 답한다
5. `edit-guard` 절차에 따라 이중검증 후 Edit 도구를 사용한다
