# 파일 편집 이중검증 규칙

이 규칙은 모든 파일 수정 작업에 자동 적용된다.

## Edit 도구 사용 전 필수 4단계

### 1단계: 왜
- 현재 코드에서 어떤 결함 또는 요구가 실제 파일에서 확인됐는가
- 추측 표현으로 이유를 설명하지 않는다

### 2단계: 어떻게
- 요청이 들어오는 위치부터 응답, 저장, 표시, 배포까지 어떤 파일 순서를 통과하는가
- 프레임워크, 언어, 폴더명은 고정하지 않는다
- 실제 진입 파일, 라우트 파일, 서비스 파일, 데이터 파일, UI 파일은 Read/Grep 결과로만 확정한다

### 3단계: 무엇을
- 어떤 필드, 함수, 타입, 테이블, 엔드포인트, 설정값이 변경 대상인가
- "이 부분"이나 "해당 로직" 같은 모호한 표현 금지
- 파일명:라인번호 기준으로 정확히 지정한다

### 4단계: 어떤 식으로
- old_string이 현재 파일에 실제로 존재하는가
- 변경 후 연관 파일과의 이름, 타입, 필드, 설정, 호출 흐름 불일치가 발생하지 않는가

## 연관 파일 전수 확인

- **진입점**: 앱 시작, 요청 진입, 명령 실행, 작업 스케줄 진입 파일을 확인한다
- **호출부**: 수정 대상 파일을 import, require, include, 호출, 참조하는 모든 파일을 확인한다
- **타입·스키마**: 타입 정의, DTO, 요청/응답 스키마, DB 스키마, 검증 스키마와 일치하는지 확인한다
- **설정**: 환경변수, ini, json, yaml, toml, env 파일의 키 이름과 로딩 경로를 확인한다
- **데이터 흐름**: 입력 → 검증 → 변환 → 저장 → 응답 → 표시 흐름이 끊기지 않는지 확인한다
- **외부 연동**: 외부 API, 큐, 파일 저장소, 다른 서비스 호출은 문서와 실제 클라이언트 코드를 대조한다

## 이중검증 체크리스트

- [ ] 수정 대상 파일을 실제로 읽었다
- [ ] old_string이 현재 파일에 정확히 존재함을 확인했다
- [ ] 연관 호출 파일을 Grep으로 찾아 읽었다
- [ ] 타입·스키마·설정 키와 일치를 확인했다
- [ ] 데이터 저장 또는 외부 연동 흐름을 확인했다
- [ ] 프로젝트 고유 경로, API 값, DB 접속값을 배포 파일에 남기지 않았다
- [ ] 가짜 구현, 더미 데이터, 임시 처리가 없다
- [ ] 인라인 중복 타입이나 중복 유틸을 만들지 않았다
- [ ] 주석으로 코드를 숨기거나 우회하지 않는다

## 절대 금지

| 금지 행동 | 이유 |
|----------|------|
| Write 도구로 기존 파일 덮어쓰기 | 기존 파일 덮어쓰기 금지 |
| 실제 읽기 없이 구조를 추측해서 Edit | 임의 판단 금지 |
| 특정 프레임워크를 파일 확인 없이 전제 | 배포 대상 프로젝트마다 구조가 다름 |
| 프로젝트 고유 API URL, DB 계정, 비밀값을 배포 파일에 유지 | 배포 패키지 오염 |
| `npm run dev/build` 같은 실행으로 오류 확인 | 금지된 실행 |
| `| head`, `| tail`, `2>&1` 포함 Bash 명령 | 출력 자르기 금지 |
| python/node/bash로 파일 직접 읽기·쓰기 | Read/Edit/Write 도구 우회 금지 |

## 완료 보고 조건

- 변경한 파일과 호출한 파일의 앞뒤 흐름이 실제 확인됐다
- 사용자 요구와 연결된 필드가 입력 → 변환 → 저장 → 표시까지 끊기지 않는다
- 프로젝트 고유 값이 예시 양식 또는 자리표시자로 교체됐다
- 금지된 빌드 없이 정적 검증을 완료했다
- 검증하지 못한 부분은 "미확인"으로 분리해서 보고했다
