이 명령은 배포 대상 프로젝트에서 파일을 수정하기 전 반드시 거쳐야 하는 이중검증 절차를 강제한다.
프레임워크와 폴더 구조를 고정하지 않는다.

## 1단계: 수정 전 필수 확인

수정 대상 파일을 변경하기 전에 다음 4가지 질문에 실제 파일 근거로 답해야 한다.

### 왜
- 현재 코드나 데이터 흐름에서 어떤 결함 또는 요구가 확인됐는가
- 사용자의 지시 원문에서 어떤 문제를 해결하라고 했는가

### 어떻게
- 요청이 들어오는 위치부터 응답 또는 저장까지 어떤 파일 순서를 통과하는가
- 앱 진입점, 라우트, 컨트롤러, 서비스, 저장소, UI, 작업자 파일을 실제 구조 기준으로 확인한다

### 무엇을
- 어떤 필드, 함수, 타입, 테이블, 엔드포인트, 설정값이 변경 대상인가
- 변경 대상의 현재 값과 변경 후 값을 명시한다

### 어떤 식으로
- old_string과 new_string을 정확히 식별했는가
- 변경 후 연관 파일과의 이름, 타입, 필드, 설정 불일치가 발생하지 않는가

## 2단계: 연관 파일 전수 확인

- 수정 대상 파일을 호출하거나 참조하는 모든 파일을 Grep으로 찾아 읽는다
- 타입, DTO, 검증 스키마, DB 스키마, 응답 스키마와 일치하는지 확인한다
- 설정 관련 수정이면 env, ini, json, yaml, toml 로더와 키 이름을 확인한다
- 외부 API 관련 수정이면 공식 문서와 실제 클라이언트 코드를 대조한다

## 3단계: Edit 도구 사용 규칙

- 기존 파일은 Edit 도구만 사용한다
- 신규 파일 생성은 사용자 요청 범위 안에서만 수행한다
- old_string은 유일하게 식별 가능한 충분한 맥락을 포함해야 한다
- 여러 파일을 수정할 때도 파일마다 위 절차를 각각 수행한다

## 완료 보고 기준

- 변경한 파일과 호출한 파일의 앞뒤 흐름이 실제 확인됐다
- 사용자 요구와 연결된 필드가 입력 → 변환 → 저장 → 표시까지 끊기지 않는다
- 금지된 빌드 또는 개발 서버 실행 없이 정적 검증을 완료했다
- 배포 파일에 개인 API 값, DB 접속값, 절대 경로가 남아 있지 않다
- 검증하지 못한 부분은 완료가 아니라 미확인 영역으로 분리해서 보고했다

## 절대 금지 행동

| 금지 행동 | 이유 |
|----------|------|
| Write 도구로 기존 파일 덮어쓰기 | 기존 파일 맥락 파괴 |
| 실제 읽기 없이 구조를 추측해서 Edit | 임의 판단 |
| 특정 프레임워크를 전제하고 수정 | 배포 대상 프로젝트마다 구조가 다름 |
| 프로젝트 고유 API URL, DB 계정, 개인 경로 유지 | 배포 패키지 오염 |
| 빌드 또는 개발 서버 실행 | 금지된 실행 |
| 출력 자르기 파이프 사용 | 오류·경고 누락 |
| 파일 조작 우회 명령 사용 | 도구 경계 위반 |
